## Architecture

### Route groups

Two distinct app surfaces share the same Next.js `[locale]` layout:

- `src/app/[locale]/(erp)/` — Back-office for authenticated business users (sidebar layout, `ErpLayout`)
- `src/app/[locale]/(storefront)/` — Public catalog + authenticated customer portal (`/portal/`)

The middleware (`middleware.ts`) handles three things in order: Supabase session refresh → next-intl locale routing → security headers. The matcher excludes `/api/*`, so API routes are never processed by intl routing.

Only `fr` locale is active. `ty` and `mq` stub files exist in `src/locales/` for future translation but are excluded from routing.

### API layer

All API route handlers are wrapped in `withAuth()` from `src/lib/auth/api-auth.ts`. This wrapper:
1. Authenticates via Supabase session cookie **or** `Authorization: Bearer <api_key>` header
2. Requires a `team_id` query param and verifies the caller belongs to that team
3. Provides a pre-built `auth.supabase` client scoped to the right auth method (RLS-respecting for sessions, service-role for API keys)

Pattern for every handler:
```typescript
export async function GET(request: NextRequest) {
  return withAuth(request, async (auth, teamId, params) => {
    requirePermission(auth, "invoices", "read");
    const { data } = await auth.supabase.from("invoices").select("*").eq("team_id", teamId);
    return NextResponse.json({ data });
  });
}
```

For API key auth, `auth.supabase` uses the service role key (bypasses RLS — authorization is already verified by `verify_api_key` RPC). For session auth, it uses the user's cookie-scoped client (RLS enforced).

### Auth & RBAC

- **ERP users**: Supabase Auth session → `team_members` table → `team_roles` with granular JSONB `permissions` (e.g. `{ "invoices": ["read", "write"] }`)
- **Storefront customers**: Separate `portal_users` table linked to Supabase Auth via `auth_user_id`. Session resolved by `getPortalSession()` in `src/lib/portal/session.ts`
- **External integrations**: Hashed Bearer API keys in `api_keys` table; validated via `verify_api_key` RPC

Owners (`is_owner = true` in `team_members`) bypass all permission checks.

### Database

18 migration files in `supabase/migrations/`. Numbering: `00001_schema_complet.sql` contains the full initial schema (migrations 00004-00026 were squashed into it). Feature migrations resume at `00027_` → `00041_`. The DB timezone is `Pacific/Tahiti`. Extensions in use: `uuid-ossp`, `vector` (pgvector), `pg_trgm`, `pgcrypto`.

All business tables have `team_id` and RLS policies scoped to it via `get_teams_for_authenticated_user()`. A custom JWT hook (`custom_access_token_hook`) injects `team_id`, `role_name`, `is_owner`, and `permissions` into the access token.

## French Polynesia Legal Compliance

This ERP is built for PF businesses. The following rules are hard requirements — never relax them.

### Invoices
- **Sequential numbering**: Generated by `generate_next_invoice_number` RPC — never manually assign
- **Immutability**: Once status moves past `draft`, the invoice number and line items cannot change
- **Soft-delete only**: Use `deleted_at` timestamp. Never hard-delete invoices. 10-year retention mandate.
- **N° TAHITI**: Must appear on both seller and buyer for every B2B invoice PDF
- **Late fee**: Fixed 5,000 F CFP (hardcoded in team settings as `late_fee_fixed`)

### TVA rates (2026)
| Rate | Description |
|------|-------------|
| 16%  | Standard (most goods) |
| 13%  | Services |
| 5%   | Essentials (food, transport, electricity) |
| 1%   | Remote archipelagos (new 2026) |
| 0%   | Exempt / export |

### Franchise en base
When `team.is_franchise_en_base = true`, invoice PDFs must hide all TVA columns and display: `"TVA non applicable, franchise en base"`

### Price display
- **Storefront (B2C)**: Always show prices **TTC** (tax-inclusive) — DGAE regulation
- **ERP / Portal (B2B)**: Show **HT** + TVA breakdown

## Design system

Brand name: **Hono**. Tone: professional, minimalist, legible.

- Shadcn/UI components live in `src/components/ui/` — use these, don't reach for raw Radix
- No purple gradients, no nested cards inside cards
- Spacing and layout create structure — avoid heavy borders everywhere
- One purposeful primary color; restrained neutral palette

## Key env vars

See `.env.example`. Critical ones not obvious from the name:
- `SUPABASE_SERVICE_ROLE_KEY` — server-only, used only for API key auth in `src/lib/auth/api-auth.ts`. Never prefix with `NEXT_PUBLIC_`
- `NEXT_PUBLIC_DEFAULT_TEAM_ID` — the team UUID that receives storefront quote requests
